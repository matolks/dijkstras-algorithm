Instructional Guide

How to Find the Shortest Path Using Dijkstra's Algorithm

Use this guide to manually find the shortest path between two locations using a weighted graph. This same process is how navigation systems compare roads, travel time, and different routes before reccomending a path.

Before You Start

Purpose

Find the shortest path from a starting node to a destination node by updating known route travel-costs until the destination is finalized.

Estimated Time

Around 10-15 minutes for a first attempt. Then roughly 5 minutes after the process becomes familiar.

Workspace Setup

Keep the graph and table visible at the same time. Each value in the table must come from a specific edge on the graph, so seeing both at once prevents incorrect updates.

Important limitation: Use Dijkstra's Algorithm only when every edge weight is positive. Negative weights will make this process give the wrong answer.

The six steps below preview the whole process before going into more detail. You start by setting up the graph, then record starting distances. Next, you repeatedly choose the smallest current distance of unvisited node and update nearby nodes. Continue until the destination is finalized and trace the finished route.

Materials and Tools

These materials and tools are necessary because Dijkstra's Algorithm is a tracking procedure. The weighted graph shows which moves are possible/available, the distance table keeps track of the current travel-costs found so far, and the visited node mark prevents you from revisiting nodes that have already been finalized.

Key Terms Used in the Steps

These terms are used directly in the procedure and game below.

NodeA point in the graph. In a road network, a node represents an intersection.
EdgeA connection between two nodes. In a road network, an edge represents a road.
WeightThe cost of using an edge. In a road network this would represent time, distance, or traffic delay.
Weighted GraphA structure made up of nodes and edges with weights to represent connections and relationships between places.
Starting NodeThe node you start at. AKA your current locatio.n
Destination nodeThe node you are trying to get to. AKA where you are going.
Travel-CostThe cost of the edges added up to get to your current position.
Current distanceThe best known total travel cost from the start node to another node so far.
Shortest PathThe route using the least travel-cost to get to the destination, or node
Visited nodeA node whose shortest path has been finalized.
Previous nodeThe node used immediately before the current node on the best known route.

1 Create and initialize the table.

Make a table with every node listed out and have a column for current distance, previous node, and status. To initizalize, identify A as the starting node and E as the destination node. Then set A's current distance to 0, set every other current distance to ∞, since their current distance is unknown. Leave previous node entries blank, and all other statuses as unvisted.

Table update: A = 0       B, C, D, and E = ∞.

A is the only node with a known distance at the start.

Prevent a common mistake: Do not choose the smallest single edge. Choose the unvisited node with the smallest current distance from A.

2 Visit A and update B and C.

Mark A as visited. Look at the edges leaving A in Figure 1 and add A's distance, 0, to each edge weight.

  • A → B costs 0 + 4 = 4. Set B's current distance to 4 and previous node to A.
  • A → C costs 0 + 2 = 2. Set C's current distance to 2 and previous node to A.

Table update: B changes from ∞ to 4. C changes from ∞ to 2.

The smallest current distance of unvisted nodes is now C = 2.

3 Visit C and update D.

Choose C because C = 2 is smaller than B = 4. Look at the edges leaving C and add C's distance each edge.

  • C → D costs 2 + 8 = 10. Set D's current distance to 10 and previous node to C.

Table update: D changes from ∞ to 10 through C.

The smallest current distance of unvisted nodes is now B = 4.

4 Visit B and revise D and E.

Choose B because B = 4 is smaller than D = 10. Look at the edges leaving B and add B's distance each edge.

  • B → D costs 4 + 5 = 9. Since 9 is less than D's current 10, change D to 9 and previous node to B.
  • B → E costs 4 + 10 = 14. Eet E's current distance to 14 and previous node to B.

Table update: D changes from 10 to 9. E changes from ∞ to 14.

This is a key correction step. Replace a longer route when a shorter route is found.

5 Visit D and revise E.

Choose D because D = 9 is smaller than E = 14. Look at the edges leaving D and add D's distance each edge.

  • D → E costs 9 + 2 = 11. Since 11 is less than E's current 14, change E to 11 and previous node to D.

Table update: E changes from 14 to 11.

E is now has the smallest current distance.

6 Stop when E is selected.

Select E. Stop because the destination has the smallest current distance among all unvisited nodes.

Table update: E is marked destination reached with a final distance of 11.

The smallest travel-cost from A to E is 11.

7 Trace the final route backward.

Use the previous node column to trace from E back to A.

  • E came from D.
  • D came from B.
  • B came from A.

Final Path: A → B → D → E. Total travel-cost: 11.

Troubleshooting / FAQ

Do I have to create a table?

For a manual instructional set, yes. The table is the easiest way to track current distances, previous nodes, and visited nodes without relying on memory.

What if two unvisited nodes have the same smallest current distance?

Choose either one. If the distances are equal, both selections are valid as long as you keep updating neighbors correctly.

What if a new path to a node is longer than the current value?

Ignore the longer path. Only change a current distance when the new travel-cost is smaller.

Why do I need the previous node column?

The distance column gives the travel-cost. The previous node column gives the path that produced that cost.

Can I stop before visiting every node?

Yes. If you only need one destination, stop when that destination is selected as the smallest current distance unvisited node.

Glossary

Dijkstra's Algorithm
A shortest path algorithm that finds the smallest travel-cost path from a starting node to a destination node on a weighted graph with positive weights.
Weighted Graph
A structure made of nodes and edges with weights that represents relationships or connections.
Shortest path
The path with the smallest total travel-cost, not guarenteed to be the fewest number of edges.
Infinity (∞)
A placeholder meaning that no path to that node has been found yet.
Total cost
The sum of all edge weights along a path.
